# 1. DT COFAS Pallet Inserter System Server 개발

**기간:** 2023.07 ~ 2023.10

**기술 환경:** Java 8, Spring Boot 2.6.7, Maven  
**통신 프로토콜:** OMRON FINS (TCP/IP)

**적용 설비:**  
OMRON PLC, Pallet Inserter, Conveyor, Robot Arm, Pusher, Stopper

---

## 1.1 프로젝트 개요

스마트 팩토리 환경에서 **Pallet Inserter 및 물류 이송 라인을 통합 제어하는 실시간 설비 제어 서버**를 설계 및 개발하였습니다.

PLC 저수준 통신, 공정 제어 엔진, 외부 시스템 연동까지 전체 제어 아키텍처를 설계하여 설비 제어 로직을 서버 기반으로 통합 관리하도록 구축하였습니다.

해당 시스템은 **러시아 수출 프로젝트 설비에 적용되어 약 1억 원 규모 설비 제어 서버로 상용 납품**되었습니다.

---

## 1.2 PLC 통신 설계

OMRON FINS TCP/IP 소켓 기반 메모리 영역 Read/Write 처리를 구현하여 실시간 설비 데이터 통신 구조를 구축하였습니다.

또한 **Heartbeat 기반 연결 상태 감시 및 자동 재연결 로직**을 적용하여 네트워크 단절 상황에서도 안정적으로 통신이 복구되도록 설계하였습니다.

통신 계층과 공정 제어 계층을 분리하여 시스템 구조의 안정성과 유지보수성을 확보하였습니다.

주요 아키텍처 구성

1. 네트워크 계층 (NetworkRunner)

PLC와 TCP/IP 기반 통신 수행 (OMRON FINS 프로토콜)

PLC 메모리 영역 512 Address 단위 읽기/쓰기 처리

2. 제어 계층 (ControlRunner)

설비별 상태 머신(State Machine) 기반 제어 로직 실행

Pallet Inserter, Robot Arm, Conveyor, Stopper 등 공정 장비 제어

---

## 1.3 상태 머신 기반 공정 제어 엔진 구축

설비 공정 제어를 위해 **Stage 단위 상태 전이(State Machine) 기반 Control Runner 구조**를 설계하였습니다.

자동/수동 모드를 분리하여 운영 시나리오별 제어 안정성을 확보하였으며,  
Inserter 진입 → 이송 → 배출 단계별 제어 로직과 Robot Arm, Stopper, Pusher 설비 연동 로직을 구현하였습니다.

또한 **비동기 명령 처리 구조**를 적용하여 실시간 제어 응답성을 개선하였습니다.

---

## 1.4 PLC 데이터 모델링 및 외부 연동 인터페이스 구축

PLC Raw Memory 데이터를 도메인 객체로 변환하는 **Packet Mapping Layer**를 설계하여 수백 개 PLC 메모리 주소를 구조적으로 관리할 수 있도록 모델링하였습니다.

또한 설비 상태 조회 및 수동 제어를 위한 **REST API 인터페이스**를 구현하고 MES 및 모니터링 대시보드와 연동할 수 있는 서버 구조를 구축하였습니다.

---

## 1.5 성과

### 기술 성과

- PLC 통신, 공정 제어 엔진, 데이터 모델링, 외부 시스템 연동 구현
- 통신 계층 / 제어 계층 분리 구조 설계
- 상태 머신 기반 설비 제어 표준 구조 정립
- 네트워크 장애 대응을 위한 Fail-Safe 통신 구조 구현

### 사업 성과

- 러시아 설비 수출 프로젝트 적용
- 약 1억 원 규모 설비 제어 서버 상용 납품
- 해외 현장 안정적 운영

# 2. WSA MES/ERP 개발

**기간:** 2024.03 ~ 2024.12  
**역할:** 운영 시스템 분석, 구조 개선, 백오피스 고도화

**기술 환경:** Java 21, Spring Boot 3.2, Gradle Multi-module, JPA, Querydsl, MySQL, Kafka, Flyway, JWT, AES

**구성 도메인:**  
Master, Production, Purchase, Inventory  
(공통/연동 모듈: Util, Interface, Monitoring)

**아키텍처:**  
단일 모놀리스 코드베이스를 도메인 모듈로 분리하고, 게이트웨이 연동을 통해 애플리케이션 레벨 MSA 방식으로 운영

---

## 2.1 모듈별 도메인 기획 및 API 개발

### 운영 시스템 파악 및 요구사항 도출

기존에는 단순 회사 프로젝트를 CRUD 중심으로 관리하는 PMS가 운영되고 있었으나, 이를 생산 관리 기능을 포함한 MES/ERP로 전면 재개발하는 것이 목표였습니다.  
DB부터 새롭게 설계하는 방향이었기 때문에 기존 시스템 마이그레이션보다는 **도메인 이해와 요구사항 정의가 핵심 과제**였습니다.

제조 도메인에 대한 이해를 빠르게 확보하기 위해 다음과 같은 방식으로 접근했습니다.

- 기존 CRUD 기반 PMS를 생산 관리 중심 MES/ERP 시스템으로 재구축하는 프로젝트 참여

- 신규 DB 설계를 위해 제조 도메인 및 생산 관리 프로세스 분석 수행

- 사용자 인터뷰 및 타사 MES 벤치마킹을 통해 생산 공정 흐름과 핵심 데이터 모델 도출

- 분석 결과를 기반으로 시스템 설계에 필요한 요구사항 정리 및 구조 정의

---

## 2.2 모듈형 아키텍처 설계

### 아키텍처 방향 결정 배경

프로젝트 초기에는 완전한 MSA와 단일 모놀리스 구조 사이에서 아키텍처 방향을 결정해야 했습니다.

완전한 MSA는 서비스 디스커버리, 게이트웨이, 배포 자동화 등 인프라 설계가 필요했으나 프로젝트 초기 단계에서 해당 환경을 구축하기에는 **초기 비용과 복잡도가 높다는 문제**가 있었습니다.

반면 단일 모놀리스 구조는 하나의 서버가 모든 책임을 담당하게 되어 **장애 격리 및 운영 유연성이 떨어지는 문제**가 있었습니다.

이에 따라 다음과 같은 **중간 전략을 선택하였습니다.**

- 데이터베이스는 단일 구조 유지
- 코드베이스는 도메인 단위 멀티 모듈 구조로 분리

모듈 구조

- master
- production
- purchase
- inventory

`master` 모듈에 공통 엔티티와 유틸을 배치하고 각 도메인 모듈이 이를 의존하는 구조로 설계하여 도메인 간 경계를 유지하면서 개발 효율성을 확보하였습니다.

---

## 2.3 모듈 별 비즈니스 로직 설계

주요 비즈니스 로직을 도메인 단위 모듈로 분리하여 관리하도록 구성하였습니다.

---

### Production (생산 관리 모듈)

생산 관리 모듈은 프로젝트 아이템이 생산 가능 상태로 확정된 이후 실제 제조 수행을 위한 공정 흐름을 생성하고, 작업 지시 및 작업 실적을 관리하는 기능을 담당합니다.

생산 대상 품목의 **공정 라우팅(Route)** 정보를 기반으로 작업 단위를 생성하고 각 작업의 상태 및 실적을 기록하여 생산 이력 추적이 가능하도록 설계하였습니다.

#### 주요 기능

- 프로젝트 아이템 생산 확정
- 공정 라우팅 기반 작업 흐름 생성
- 공정별 작업 지시 생성
- 작업 수행 및 상태 관리
- 작업 실적 등록 및 생산 상태 반영

#### 생산 흐름

생산 확정 → 공정 라우팅 생성 → 작업 지시 발행 → 작업 수행 → 작업 실적 반영

#### 기술적 과제 및 해결

생산 공정 흐름과 품질 관리 기능을 연계하는 과정에서 공정 간 불량 처리 로직 구현이 주요 기술적 과제였습니다.

생산 공정은 다음과 같은 구조를 가집니다.

A → B → C → D

특정 공정(C)에서 불량이 발생하면 단순히 해당 공정에만 기록되는 것이 아니라 이전 공정(A~C) 이력에도 불량 수량이 반영되어야 하는 요구사항이 존재했습니다.

또한 하나의 품목은 여러 하위 부품으로 구성되며 각 부품은 독립적인 공정 라우팅을 가지기 때문에 단순 선형 탐색으로는 영향 범위를 정확하게 계산하기 어려웠습니다.

이를 해결하기 위해 공정 라우팅을 BFS 알고리즘을 활용해 품질 검사 지점에서 재작업 목표 라우팅까지의 최단 경로를 탐색하도록 구현했습니다.

경로 탐색 이후는 경로를 순차적으로 순회하면서 각 노드(라우팅) 마다 이전 노드와의 관계를 순차 관계인지 계층 관계인지에 따라 부품 구성 비율을 계산 하였습니다.

또한 불량 처리 이후 품질 관리 → 생산 이력 반영 → 재고 수량 반영의 단계적 처리 흐름을 구성하여, 각 단계가 순차적으로 처리되도록 설계함으로써 여러 도메인(품질, 생산, 재고)에 걸친 데이터 정합성을 유지할 수 있도록 분산 트랜잭션 흐름을 관리했습니다.

이를 통해 다음과 같은 효과를 얻을 수 있었습니다.

- 공정 간 불량 영향 범위 자동 계산

- 다단계 공정 구조 대응

- 품질·생산·재고 데이터 간 정합성 유지

- 공정 불량 발생 시 선행 공정 이력 및 재고 반영 자동화

---

### Inventory (재고 관리 모듈)

재고 관리 모듈은 입고, 출고, 이동, 조정 과정을 관리하고 현재 재고 상태를 실시간으로 확인할 수 있도록 설계된 도메인입니다.

생산, 구매, 품질 등 다른 업무 모듈과 연계되어 자재 및 완제품의 수량 변동을 반영하며 재고 이력과 현재 수량의 정합성을 유지하는 역할을 담당합니다.

#### 주요 기능

- 품목 관리
- 입고 처리
- 출고 처리
- 재고 이동
- 재고 조정
- 재고 상태 조회

#### 설계 의도

재고 관리 모듈은 단순 수량 저장이 아니라 **입고 / 출고 / 이동 / 조정 이력을 함께 관리하는 구조**로 설계되었습니다.

이를 통해

- 재고 변동 이력 추적
- 재고 차이 원인 분석
- 생산 / 구매 / 출하 도메인과 데이터 정합성 유지

를 달성할 수 있도록 설계하였습니다.

#### 기술적 과제 및 해결

재고 입고 처리 과정에서 **트랜잭션 및 동시성 제어 문제**가 발생했습니다.

재고 로직은 `Result` 추상 엔티티 기반으로 재고 변동 이력을 생성하도록 설계되어 있었으며 해당 API는 여러 서비스에서 호출되는 구조였습니다.

이로 인해 동일 트랜잭션 내에서 재고 로직이 수행되며 **DB Lock Wait** 문제가 발생했습니다.

이를 해결하기 위해

- 재고 반영 로직을 `afterCommit` 이벤트 방식으로 분리
- 비즈니스 트랜잭션과 재고 반영 시점 분리

구조로 변경하였습니다.

또한 동시 출고 요청으로 인한 재고 정합성 문제를 방지하기 위해 **원자적 조건부 업데이트 방식**을 적용하였습니다.

예시

UPDATE stock
SET quantity = quantity - :qty
WHERE item_id = :id
AND quantity >= :qty

이를 통해

- 동시 출고 상황에서도 하나의 트랜잭션만 성공
- 재고 음수 방지
- 동시성 안정성 확보

를 달성할 수 있었습니다.

---

### Purchase (구매 관리 모듈)

구매 관리 모듈은 외부 공급업체와 자재 수급 과정을 관리하는 도메인입니다.  
프로젝트 및 생산 계획에 필요한 자재를 적시에 확보할 수 있도록 **구매 요청 → 견적 → 발주 → 구매 실적 등록**의 전체 흐름을 관리하도록 설계되었습니다.

또한 공급업체와 품목 정보를 연계하여 구매 진행 상태와 구매 이력을 추적할 수 있도록 구성하였습니다.

---

### 주요 기능

#### 구매 요청 관리

- 프로젝트 또는 자재 수요를 기반으로 구매 요청서를 생성하고 관리합니다.
- 요청 품목, 수량, 납기일 등의 정보를 등록하며 승인 및 완료 상태를 관리합니다.

#### 구매 견적 관리

- 구매 요청을 기반으로 견적서를 작성하고 공급업체별 견적 정보를 관리합니다.
- 견적 승인 및 완료 처리를 통해 이후 발주 단계로 연계할 수 있도록 구성하였습니다.

#### 구매 발주 관리

- 승인된 견적 또는 구매 요청을 기준으로 발주서를 생성하고 관리합니다.
- 발주 품목, 단가, 수량, 납기일 등의 정보를 관리하며 승인 상태를 반영합니다.

#### 외주 발주 관리

- 일반 구매 외에 외주 작업이 필요한 품목에 대해 외주 발주서를 생성하고 관리합니다.
- 외주 업체, 작업 내용, 처리 상태를 관리하여 외주 생산 프로세스를 지원합니다.

#### 구매 실적 관리

- 발주 완료 이후 실제 구매 결과를 등록하고 실적 데이터를 관리합니다.
- 구매 수량, 금액, 처리 결과 등을 기록하여 구매 이력을 추적할 수 있도록 구현하였습니다.

#### 품목-거래처 연계 관리

- 특정 품목과 공급업체 간 매핑 정보를 관리합니다.
- 품목별 구매 가능 거래처를 관리하여 구매 문서 작성 시 활용할 수 있도록 하였습니다.

#### 구매 알림 및 메일 발송

- 발주 일정 또는 만료 예정 품목에 대해 알림 및 메일 발송 기능을 제공합니다.
- 담당자가 구매 진행 상태를 사전에 확인할 수 있도록 지원합니다.

### 설계 의도

구매 관리 모듈은 단순 발주 관리가 아니라 생산 계획과 연계된 자재 수급 관리 역할을 수행하도록 설계되었습니다.  
이를 통해 생산 계획에서 필요한 자재를 기준으로 구매 요청을 생성하고 구매 진행 상태를 추적할 수 있도록 구성하였습니다.

#### 기술적 과제 및 해결

기존 시스템에서는 프로젝트를 선택하면 해당 프로젝트에 필요한 자재 수요를 쿼리를 통해 계산하여 사용자에게 보여주는 방식으로 구현되어 있었습니다.

사용자는 계산된 자재 수요와 현재까지 구매된 수량을 확인한 후 부족한 자재에 대해 구매 발주를 진행하는 구조였습니다.

그러나 운영 과정에서 **프로젝트 진행 중 구매팀의 판단에 의해 자재 수요를 변경하거나 관리할 수 있어야 한다는 요구사항**이 추가되었습니다.

기존 구조에서는 자재 수요가 단순 쿼리 계산 결과였기 때문에 사용자가 직접 수정하거나 관리할 수 없었고, 요구사항을 반영하기 어려운 구조적 한계가 있었습니다.

이를 해결하기 위해 **구매팀이 프로젝트 단위 자재 수요를 발행하는 구조로 도메인 모델을 변경**하였습니다.

프로젝트 자재 수요에 관한 엔티티를 도입하여 구매팀이 프로젝트 기준으로 필요한 자재 수요를 등록하도록 하였으며, 이후 구매 요청 및 발주 프로세스는 해당 자재 수요 데이터를 기준으로 진행되도록 설계하였습니다.

이를 통해 다음과 같은 개선을 할 수 있었습니다.

- 프로젝트 단위 자재 수요 발행 및 수정 기능 지원
- 자재 수요와 구매 발주 프로세스 간 명확한 연계
- 프로젝트 기준 자재 수요 이력 관리

결과적으로 기존 쿼리 기반 계산 방식의 한계를 해결하고 **프로젝트 자재 수요를 도메인 데이터로 관리할 수 있는 구조**로 개선하였습니다.

---

## 2.4 기존 시스템 구조 및 레거시 코드 개선

기존 시스템에서는 공통 라이브러리 의존도가 높아 구조적인 제약이 존재했으며 이를 개선하기 위한 작업을 진행하였습니다.

### 도메인 모델 구조 개선

기존 회사는 DTO를 사용하지 않고 엔티티를 직접 API 응답으로 사용하는 구조였습니다.

또한 엔티티에 `@Setter`가 광범위하게 사용되고 있어 객체 상태 변경 추적이 어려웠습니다.

이를 개선하기 위해

- DTO 기반 API 구조 도입
- 도메인 엔티티 Setter 사용 금지

구조로 변경하였습니다.

---

### 조회 구조 개선 (QueryParameter → Querydsl)

기존 시스템에서는 HTTP Query String을 JPA Specification으로 변환하는 `QueryParameter` 라이브러리를 사용하고 있었습니다.

그러나 복잡한 조회 조건 구현 시 유연성이 떨어지고 클라이언트가 엔티티 구조를 알 경우 의도하지 않은 조회 쿼리가 생성될 수 있는 문제가 존재했습니다.

처음에는 해당 라이브러리를 개선하고자 간단한 필터링의 경우느 Specification 기반 조회 후 애플리케이션 레벨에서 추가 필터링을 수행하는 방식을 고려하였으나

- 페이지 데이터 누락
- 조회 정합성 문제

가 발생했습니다.

이를 해결하기 위해 조회 로직을 **Querydsl 기반 동적 쿼리 구조로 전환**하였습니다.

---

### 공통 서비스 구조 개선

기존에는 `BaseService`, `BaseController` 기반 Generic CRUD 구조를 사용하고 있었습니다.

하지만 실제 비즈니스 로직 확장 시 도메인 로직과 공통 로직이 혼재되는 문제가 발생했습니다.

이에 따라

- 도메인 서비스 단위 구조로 변경
- BaseService 구조 제거

하여 서비스 책임을 명확하게 분리하였습니다.

---

### 결과

레거시 구조를 개선하면서

- DTO 기반 API 구조 도입
- Querydsl 기반 조회 구조 전환
- 도메인 서비스 단위 설계

를 적용하였습니다.

또한 Lazy Loading으로 인한 추가 쿼리 문제를 해결하여 대부분의 조회 API가 **100ms 수준의 응답 시간**을 유지하도록 개선하였습니다.

## 3.1 프로젝트 개요

### MSA로 전환 하게 된 이유

기존 시스템은 단일 DB를 공유하는 다중 모듈 구조로 구성되어 있었습니다.
각 모듈은 논리적으로 분리되어 있었지만 동일한 데이터베이스를 사용하고 있었기 때문에 서비스 간 데이터 결합도가 높았고, 특정 기능 변경 시 다른 모듈까지 영향을 받을 가능성이 있었습니다.

또한 일부 기능은 모듈 간 API 호출을 통해 처리되는 구조였는데, 동일 DB를 공유하는 환경에서 연관 관계 기반의 트랜잭션 처리 과정에서 락 경합과 대기 시간이 증가하는 문제가 발생했습니다.

이러한 구조는 서비스 규모가 커질수록 확장성과 운영 유연성에 제약이 될 수 있다고 판단했습니다.
따라서 서비스별 데이터 소유권을 명확히 분리하고, 서비스 단위로 독립적인 확장과 배포가 가능하도록 MSA 구조로 전환하는 방향을 검토하게 되었습니다.

## 3.2 MSA 전환 및 서비스 경계 재정의

### 서비스 경계 정의 원칙

완전한 MSA를 목표로 서비스별 DB를 완전히 분리했습니다. 인증은 모든 서비스에서 공통으로 필요하고 패스워드 관리를 독립적으로 운용하기 위해 로그인 서비스로 분리했으며, 직원/조직 정보는 여러 서비스에서 가장
빈번하게 참조되는 엔티티인 동시에 권한 정보와 결합되는 특성이 있어 직원 관리 서비스로 분리했습니다. 2번 프로젝트의 모듈 구조를 그대로 이어받지 않고 완전히 새로 설계하였습니다.

### 서비스 간 데이터 동기화 전략

MSA 전환 과정에서 각 서비스의 데이터베이스를 완전히 분리하면서, 서비스 간 데이터 참조 방식을 어떻게 설계할지가 핵심 과제였습니다.
목표는 서비스 간 API 분산 조회를 최소화하고, 각 서비스가 자체 데이터베이스만으로 독립적으로 동작할 수 있는 구조를 만드는 것이었습니다.

초기에는 AOP 기반 BFF 패턴을 적용해 컨트롤러 응답을 가로채고, 필요한 데이터를 다른 서비스의 API 호출 결과와 조합하는 방식을 검토했습니다.
그러나 이 방식은 모든 서비스가 공통 AOP 로직에 의존하게 되어 구조적 결합도가 높아졌고, 서비스 간 네트워크 홉이 증가하면서 응답 지연과 장애 전파 가능성도 함께 커지는 한계가 있었습니다.

이러한 한계를 확인한 뒤, 서비스 간 실시간 조회를 최소화하는 대신 필요한 데이터는 각 서비스가 자체적으로 보유하는 방향으로 설계를 전환했습니다.
그 과정에서 일부 데이터 중복은 허용하되, 데이터 정합성과 처리 안정성을 확보할 수 있는 구조가 더 적절하다고 판단했습니다.

최종적으로는 Outbox + Kafka Connect CDC 기반의 데이터 동기화 구조를 도입하여, 각 서비스가 자체 DB를 기준으로 독립적으로 동작할 수 있도록 설계했습니다.
도메인 변경이 발생하면 비즈니스 트랜잭션과 함께 Outbox 이벤트를 저장하고, 이후 CDC를 통해 Kafka로 이벤트를 발행하여 필요한 컨슈머 서비스에 변경 내용을 전달했습니다.
컨슈머 서비스는 사전에 정의된 헤더 타입과 페이로드 규약을 기준으로 동기화 작업을 수행하도록 구성했습니다.

또한 컨슈머 측에서는 처리 완료한 이벤트를 Inbox 테이블에 기록하여, 이미 반영된 이벤트의 중복 처리를 방지하고 멱등성을 보장하도록 설계했습니다.
이를 통해 서비스 간 직접 API 호출 의존도를 줄이고, 각 서비스가 자체 데이터베이스만으로도 안정적으로 동작할 수 있는 구조를 구현했습니다.

### 초기 데이터 동기화 및 풀 싱크 전략

이벤트 기반 데이터 동기화 구조를 구축했지만, 모든 상황을 이벤트만으로 처리할 수는 없었습니다.

특히 신규 서비스가 추가되는 경우 기존 데이터를 이벤트만으로 재구성하는 것이 불가능하기 때문에, 전체 데이터를 기준으로 동기화를 수행하는 풀 싱크(Full Sync) 작업이 필요했습니다.

또한 운영 과정에서도 데이터 정합성을 검증하거나 누락된 이벤트를 보정하기 위해 주기적인 전체 동기화 작업을 수행할 수 있는 구조를 마련하는 것이 필요했습니다.

초기에는 각 모듈의 서비스 계층에서 직접 동기화 로직을 구현하는 방식으로 작업을 진행했습니다.
그러나 이 방식은 서비스 로직이 과도하게 비대해지고, 개발자마다 구현 방식과 코드 구조가 달라 가독성과 유지보수성이 떨어지는 문제가 발생했습니다.

이를 해결하기 위해 일관된 배치 처리 구조가 필요하다고 판단했고 Spring Batch를 도입했습니다.

Spring Batch의 Chunk 기반 처리 구조를 활용하여 동기화 작업을 다음과 같은 단계로 분리했습니다.

Reader : 동기화 대상 데이터를 조회

Processor : 이벤트 규격 또는 동기화 포맷으로 데이터 변환

Writer : 대상 서비스 또는 저장소로 데이터 반영

이 구조를 적용하면서 다음과 같은 효과를 얻을 수 있었습니다.

동기화 로직을 표준화된 Reader / Processor / Writer 구조로 통일

각 서비스에서 구현하던 동기화 로직을 배치 Job 단위로 분리하여 가독성 개선

Job 실행 이력 및 처리 상태 관리를 통한 운영 안정성 확보

대량 데이터 처리가 요구되는 환경은 아니었지만, Spring Batch가 제공하는 명확한 처리 단계 구조와 실행 이력 관리 기능이 동기화 작업의 유지보수성과 운영 관리 측면에서 유리하다고 판단하여 적용했습니다.

---

## 3.3 Gateway 서비스 구축

Spring Cloud Gateway를 MSA의 중앙 진입점으로 설계하여 인증, 라우팅, 관측 기능을 게이트웨이 계층에 집중시켰습니다.
서비스별 중복 인증 로직과 개별 라우팅 관리 부담을 제거하고, 운영 환경에서의 추적성과 안정성을 강화했습니다.

### 3.3.1 동적 라우팅(Consul 연동) 설계

- Consul 서비스 디스커버리 기반으로 `특정이름-*(해당 서비스 이름)` 서비스 라우트를 자동 생성

- 서비스 등록/해제 이벤트(Heartbeat) 감지 시 라우트 증분 반영 및 `RefreshRoutesEvent` 발행

- 서비스 네이밍 표준 규칙 적용으로 신규 서비스 온보딩 자동화

- 삭제된 서비스의 잔존 라우트(Orphan Route) 정리 로직 구현으로 오라우팅 방지

**성과**

- 수동 라우팅 설정 제거로 운영/배포 리드타임 단축

- 서비스 스케일 인/아웃 및 재기동 상황에서도 라우팅 안정성 확보

### 3.3.2 전역 인증 필터(JWT/Refresh) 구현

- Global Filter에서 요청 선처리 후 공개 경로(`auth`, `actuator` 등)만 예외 처리

- `Authorization`/`Refresh` 쿠키 추출 후 Auth 서비스 검증 위임

- 인증 성공 시 사용자 정보를 내부 헤더로 주입하여 하위 서비스 인증 컨텍스트 표준화

- Refresh 요청은 별도 분기 처리하여 일반 API 인증 흐름과 분리

- 외부 요청의 내부 전용 헤더 위조를 방지하기 위한 요청 헤더 정제(sanitize) 적용

**성과**

- 인증 정책 중앙화로 서비스 간 보안 일관성 확보

- 토큰 만료/재발급 시나리오 처리 표준화

- 인증 우회 가능성 감소 및 운영 신뢰성 향상

### 3.3.3 공통 응답/예외 표준화

- 인증 실패 및 예외 응답을 `ProblemDetail` 포맷으로 통일

- 상태코드/상세 메시지/요청 경로/발생 시각을 표준 필드로 제공

- 인증 서비스 호출 실패(WebClient 오류, 네트워크 예외 등)를 유형별로 매핑 처리

- 예외 상황에서도 fallback 응답을 제공해 포맷 일관성 유지

**성과**

- 프론트/백오피스 오류 처리 로직 단순화

### 3.3.4 기술적 기여 요약

- 게이트웨이를 단순 프록시가 아닌 인증/정책/관측의 공통 인프라 레이어로 고도화

- 동적 라우팅 + 전역 인증 + 표준 예외 처리 조합으로 MSA 진입 규약 수립

- 도메인 서비스가 비즈니스 로직에 집중할 수 있는 구조적 기반 제공

## 3.4 Login(Auth) 서비스 구축

단순 세션 방식 대신 JWT 기반 인증 체계를 설계하여 Access Token(단기) + Refresh Token(장기) 이중 구조를 표준화했습니다.  
로그인/재발급/로그아웃/게이트웨이 검증 흐름을 Auth 서비스에 집중시켜 인증 정책의 일관성과 운영 추적성을 높였습니다.

### 3.4.1 Access / Refresh Token 구조 설계

- Access Token에는 사용자 식별자와 최소 사용자 컨텍스트를 포함해 게이트웨이/하위 서비스가 빠르게 인증 문맥을 복원하도록 설계

- Refresh Token은 재발급 전용으로 분리하고, Access Token보다 긴 만료 시간으로 사용자 경험을 유지

- 토큰에 `tokenType` 클레임을 명시해 용도 오남용(Refresh를 API 호출에 직접 사용 등)을 방지

- 발급자(issuer), 대상(audience), 만료 시간(exp) 검증 규칙을 통일해 서비스 간 해석 차이를 제거

**성과**

- 토큰 역할 분리로 보안성과 UX를 동시에 확보

- 인증 스펙이 명확해져 게이트웨이/도메인 서비스 연동 비용 절감

### 3.4.2 Redis 기반 토큰 관리

- 로그인 시 Refresh Token을 Redis에 사용자 키 기준으로 저장하여 서버 측 세션 제어 지점을 확보

- 재발급 시 Redis 저장값과 클라이언트 제출값을 비교해 탈취/재사용 시도를 탐지

- 토큰 만료 정책과 Redis TTL을 동일한 기준으로 운영해 만료 일관성을 유지

- 다중 인스턴스 환경에서도 중앙 저장소(Redis) 기반으로 동일한 토큰 상태를 공유

**성과**

- Stateless JWT의 단점(서버 강제 만료 어려움)을 Redis로 보완

- 서버 증설/축소 상황에서도 인증 상태 일관성 확보

### 3.4.3 토큰 무효화 및 재발급 정책

- 로그아웃 시 Redis Refresh Token 즉시 삭제 + 쿠키 제거로 세션 종료를 명시적으로 처리

- 재발급 시 RTR(Refresh Token Rotation) 정책을 적용해 기존 Refresh Token을 폐기하고 신규 토큰으로 교체

- Redis 저장 토큰과 불일치 시 즉시 거부하고 필요 시 해당 사용자 토큰 체인을 정리해 재사용 공격을 차단

- 비정상 토큰(만료/서명 오류/형식 오류)은 표준 예외 코드로 매핑해 클라이언트 대응을 단순화

**성과**

- 토큰 탈취/재사용 리스크를 실무 운영 가능한 수준으로 감소

- 재로그인 유도 정책과 장애 대응 절차를 표준화

### 3.4.4 Gateway 연계 인증 처리

- 게이트웨이는 Auth 검증 엔드포인트를 통해 토큰 유효성 및 사용자 식별 정보를 확인

- 인증 성공 시 사용자 정보를 내부 전달 헤더로 주입해 하위 서비스에서 중복 인증 로직 없이 인가 처리 가능

- 공개 경로/보호 경로를 분리해 인증 비용을 최적화하고 운영 로그 추적성을 강화

**성과**

- 인증 책임 중앙화로 도메인 서비스의 비즈니스 집중도 향상

- 서비스 간 보안 정책 편차 감소

## 3.5 EMS(직원/권한) 서비스 구축

도메인 모델은 `Employee`, `Affiliation(Organization/Department)`, `Role`, `EmployeeRoleLink` 중심으로 재구성했습니다.  
조직은 `OrganizationLink`로 트리 관계를 분리해 루트 단일성 검증, 부모 변경 시 사이클 방지(DFS), 삭제 제약(자식/부서/직원 참조 존재 시 차단)을 서비스 규칙으로 명시했습니다.  
직원은 생성/수정 외에 퇴사(`active=false`, `separationDate`)와 원복 시나리오를 표준 흐름으로 정의해 운영 케이스를 도메인 행위로 흡수했습니다.

권한은 "역할-API"를 메타데이터로 관리하는 방식으로 설계했습니다.  
역할 기본 정보는 RDB(`Role`)에서 관리하고, 서비스별 API 권한 매트릭스는 MongoDB(`role_permissions`)에 분리 저장해 조회/변경 책임을 분리했습니다.  
또한 컨트롤러 메서드에 `@ApiPermission`을 선언하고, 애플리케이션 기동 시 애플리케이션 러너 단계에서 스캐너가 경로/메서드/설명을 자동 수집해 `msa_api_permissions`를 서비스 단위로 전체 교체 저장하도록 구성해
수작업 권한 등록 누락을 줄였습니다.

서비스 간 동기화는 위에서 언급한 "비즈니스 처리 + Outbox 적재" 구조로 구현했습니다.  
직원/조직/부서 변경 시점에 `OutBoxEvent(entity, eventId, eventType, payload)`를 같은 트랜잭션 안에서 기록하고, 이벤트 타입은
`CREATE/UPDATE/DELETE/CREATE_BATCH` 등 각 상황에 맞게 표준화했습니다.  
이후 Kafka Connect CDC가 Outbox를 읽어 타 서비스로 전달하도록 설계해, 애플리케이션 코드에서 직접 전송 책임을 분리하고 데이터 정합성을 안정화했습니다.

**성과**

- 직원/조직 변경 규칙을 도메인 서비스로 내재화해 운영 예외 대응 속도 향상

- 권한 메타데이터 자동 등록 체계로 배포 시 권한 누락 리스크 예방

- Outbox + Kafka Connect 기반 동기화로 서비스 간 최종 정합성 운영 모델 정착

## 3.6 LMS(연차/근태) 서비스 구축

### 서비스 목적

LMS는 MSA 인프라(Gateway, Auth, EMS)가 실제로 잘 동작하는지 검증하는 파일럿 서비스로 개발했습니다. 독립 DB와 Kafka 기반 동기화 구조가 실제 비즈니스 서비스에서 어떻게 동작하는지 확인하는
것이 주요 목적이었습니다.

### EMS 이벤트 기반 동기화

Kafka를 통해 동기화된 자체 DB에서 처리합니다. EMS의 Outbox + Kafka Connect CDC 파이프라인이 `Employee-topic`, `Department-topic`,
`Organization-topic`으로 이벤트를 발행하면, LMS의 `KafkaControlListener`가 `action` 헤더로 이벤트 타입을 분기해 처리합니다. `CREATE`, `UPDATE`,
`RESIGN`, `RESIGN_REVERT` 등 EMS에서 정의한 이벤트 타입을 그대로 매핑해 직원 생성/수정/퇴사/복직 시나리오를 모두 처리하도록 구현했습니다.

LMS는 수신한 이벤트를 `InboxEvent` 엔티티에 기록해 처리 이력을 관리합니다. EMS의 `OutBoxEvent`마다 고유한 `eventId`가 부여되어 있고, LMS는 메시지 수신 시 해당
`eventId`가 `InboxEvent`에 이미 존재하는지 확인합니다. 중복 수신된 이벤트는 스킵하고 신규 이벤트만 처리 후 `InboxEvent`에 기록하는 방식으로 멱등성을 보장했습니다. Kafka의
at-least-once 전달 특성상 동일 메시지가 재전송될 수 있는 상황에서도 데이터 정합성을 유지할 수 있습니다.

### 연차 도메인 설계

`VacationInfo`를 연도별 직원 휴가 정보의 루트 엔티티로 설계하고, `LeaveOccurrenceHistory`(발생 이력), `LeaveUsageHistory`(사용 이력),
`LeaveCarriedDetail`(이월 내역)을 하위에 두는 이력 기반 구조로 설계했습니다.

## 3.7 공통화 라이브러리 개발

MSA 구조에서 서비스가 완전히 분리되다 보니, 인증 헤더 파싱, 예외 처리, 로깅 같은 공통 코드를 서비스마다 반복 작성해야 하는 문제가 생겼습니다. 이를 해소하기 위해 공통 모듈을 GitHub Packages
기반 라이브러리로 추출하고 각 서비스가 의존성으로 가져다 쓸 수 있도록 구성했습니다.

**인증 컨텍스트 필터**는 Gateway에서 주입한 `employee` 헤더를 Base64 디코딩해 `EmployeeContext`에 담고 MDC에 사용자 정보를 세팅하는 필터를 추상 클래스로 추출했습니다. 각
서비스는 이를 상속받아 서비스별 커스터마이징만 추가하면 되는 구조로 표준화했습니다.

**요청/응답 로깅**은 MDC 기반으로 요청 URL, 메서드, 바디, 응답 상태코드, 처리 시간, 성능 카테고리를 구조화해서 기록하는 로깅 서비스를 공통화했습니다. 응답 상태에 따라 로그 레벨을 자동 분기(
5xx → error, 4xx → warn, 그 외 → info)하도록 설계하고, ELK 연동을 염두에 두고 MDC 필드를 표준화했습니다.

**예외 처리** 표준 포맷도 라이브러리에 포함해 서비스 간 오류 응답 구조를 통일했습니다.

**성과**

- 서비스별 반복 구현 제거로 개발 비용 절감

- 인증 컨텍스트와 로그 포맷 표준화로 서비스 간 일관성 확보

- 신규 서비스 추가 시 라이브러리 의존성 추가만으로 공통 기능 즉시 적용

## 3.8 배포 자동화 공통 워크플로우 개발

기존에는 서버에 직접 접속해 수동으로 빌드하고 배포하는 방식으로 운영했습니다. MSA 전환으로 서비스 수가 늘어나면서 수동 배포의 비용과 실수 가능성이 커졌고, 서비스별 자동화 배포 체계가 필요해졌습니다.

초기에는 GitHub Actions + Docker Compose 기반으로 서비스별 워크플로우를 각각 작성했습니다. 빌드, 이미지 정리, 컨테이너 기동, 헬스체크, 배포 알림 단계가 서비스마다 반복되면서 중복이
많아졌고, 한 단계를 수정하면 모든 서비스의 워크플로우를 각각 수정해야 하는 유지보수 문제가 생겼습니다.

이를 해소하기 위해 공통 워크플로우를 추출하고 서비스별 워크플로우에서 재사용하는 구조로 개선했습니다. 빌드 → 기존 컨테이너 정리 → 이미지 빌드 및 기동 → 헬스체크(`/actuator/health`) → 배포
완료 알림 흐름을 표준화하여, 신규 서비스 추가 시 서비스별 파라미터만 넘기면 동일한 배포 흐름이 적용되도록 했습니다.

**성과**

- 수동 배포 제거로 배포 실수 가능성 감소

- 공통 워크플로우 표준화로 신규 서비스 온보딩 비용 절감

- 헬스체크 자동화로 배포 후 이상 상태 즉시 감지
