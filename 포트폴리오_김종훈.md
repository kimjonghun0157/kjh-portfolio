### 1) PLC 통신 스택 직접 구현

OMRON FINS 프로토콜 클라이언트를 직접 구현했습니다. 기성 라이브러리 없이 TCP/IP 소켓 기반 메모리 영역 Read/Write를 실시간 처리하도록 설계했고, 다음 두 가지를 핵심 요구사항으로 잡았습니다.

- **구조적 분리**: 통신 계층과 제어 계층을 명확히 분리해 유지보수성 확보

### 2) 상태 머신 기반 공정 제어 엔진

Control Runner + Stage 단위 상태 전이 구조로 공정 제어 엔진을 설계했습니다.

- 자동/수동 모드를 분리해 운영 시나리오별 제어 안정성 확보
- Inserter 진입 → 이송 → 배출 단계별 로직 + 로봇 암, 스토퍼, 푸셔 연동
- 비동기 명령 처리 구조로 실시간 응답성 강화

### 3) PLC 데이터 모델링 및 외부 연동

수백 개 PLC 메모리 주소를 구조화해서 도메인 객체로 매핑하는 **Packet Mapping Layer**를 설계했습니다. 이를 통해 Raw Memory 데이터가 의미 있는 도메인 객체로 변환되어 상위 레이어에서 다루기 쉬운 구조를 만들었습니다. MES 및 대시보드 연동을 위한 REST API 인터페이스도 함께 구현했습니다.

## 💡 핵심 문제 해결

### 문제 1 — 조회 성능 및 결과 정확성 (Specification → Querydsl 전환)

**배경**

기존 시스템은 사내 공통 라이브러리(`QueryParameter`)로 동적 조회를 처리하고 있었습니다. 단순 컬럼 기반 필터링에는 동작했지만, 생산 공정 통계 화면 개발 과정에서 구조적 한계가 드러났습니다.

`Routing` 도메인의 진행률, 순수 완료 수량, 생산 가능 수량 등 주요 지표들은 여러 연관 엔티티를 조합해 계산하는 getter 메서드였고, Specification은 DB 컬럼 경로(Path) 기반으로만 Predicate를 생성하는 구조라 이런 계산 기반 필터링을 SQL로 표현할 수 없었습니다.

**임시 우회의 부작용**

어쩔 수 없이 Specification으로 1차 필터링 후 Java Stream으로 계산 필터링하는 방식을 택했는데, 근본적인 문제가 있었습니다. Specification 단계에서 이미 페이지네이션이 적용된 상태라, 이후 추가 필터링 시 페이지당 반환 건수가 들쭉날쭉해졌습니다. 다음 페이지에 조건을 만족하는 데이터가 있어도 이미 페이지 범위 밖으로 밀려 누락되고, 집계 수치도 전체 모수가 달라져 신뢰할 수 없는 상태가 되었습니다.

**해결**

리스트 조회 전반을 Querydsl 기반 동적 쿼리로 전환하고, 계산식은 `JPAExpressions` 서브쿼리로 SQL 레벨에서 처리하도록 재설계했습니다. 계산 구조가 복잡한 라우팅 통계는 별도 뷰 테이블을 구성하고 Querydsl에서 조회하는 방식으로 복잡도를 분리했습니다.

**결과**
- ✅ 조회 응답시간 **200ms 이상 → 100ms 미만** 으로 단축
- ✅ 페이지네이션 + 필터링을 DB 레벨에서 통합 처리 → 집계 오차 및 페이지 불일치 해소
- ✅ 타입 세이프 쿼리로 조회 조건 변경 시 컴파일 타임 오류 감지 → 유지보수성 향상

---

### 문제 2 — 아키텍처 방향 결정 (모듈형 중간 전략 선택)

**배경**

프로젝트 초기 완전한 MSA와 단일 모놀리스 사이에서 방향을 결정해야 했습니다.

- 완전한 MSA: 서비스 디스커버리, 게이트웨이, 배포 자동화 등 인프라 기획이 충분하지 않은 상태에서 초기 셋업 비용이 너무 큼
- 단일 모놀리스: 하나의 서버가 너무 많은 책임을 지게 되어 장애 격리와 운영 유연성이 떨어짐

**선택한 전략**

DB는 단일로 유지하되, 코드베이스를 `core / master / production / purchase / inventory` 도메인 모듈로 분리하는 중간 전략을 선택했습니다. `core` 모듈에 전체 엔티티와 공통 유틸을 집중시키고 각 도메인 모듈이 core를 의존하는 구조로, 도메인 간 엔티티 참조 문제 없이 서비스 경계를 명확히 할 수 있었습니다.

이 구조는 이후 3번 프로젝트 MSA 전환 시 서비스 경계가 모듈 단위로 이미 정의되어 분리 비용을 크게 줄이는 기반이 되었습니다.

**추가 구조 개선**

- Gateway 인증 헤더 기반 인터셉터 적용
- JWT 토큰 체계 도입
- AES 기반 개인정보 암호화 정책 적용

### 문제 3 — 공통 추상화의 한계 인식 (BaseService 설계와 철회)

빠른 개발을 위해 `BaseService<E extends BaseEntity>` 기반의 공통 CRUD 템플릿을 설계했습니다. `before()` / `after()` 훅으로 도메인별 커스터마이징 포인트를 열어뒀고, 초기 기능 개발 속도를 높이는 데 기여했습니다.

그러나 도메인 복잡도가 높아질수록 추상화가 오히려 확장을 제약하는 문제가 드러났습니다. 상태 전이와 복합 비즈니스 로직이 필요한 도메인에서는 BaseService를 우회하거나 사용하지 않는 케이스가 늘었습니다.

이를 통해 **범용 추상화보다 도메인별 서비스를 독립적으로 설계하는 방식이 더 적합**하다는 판단에 이르렀고, 이 경험은 이후 MSA 전환 시 각 서비스의 도메인 로직을 독립 설계하는 데 직접적인 교훈이 되었습니다.

### 문제 1 — 서비스 간 데이터 동기화 전략

**배경**

DB 완전 분리로 인해 서비스 간 데이터 참조 방식이 핵심 과제였습니다. 초기에는 AOP 기반 BFF 패턴을 직접 구현해 컨트롤러 응답을 가로채서 타 서비스에 추가 API 호출을 조합하는 방식을 시도했습니다. Page 단위 배치 호출로 아이템 수 문제는 완화했지만, 서비스 간 네트워크 홉 추가로 인한 API 지연은 구조적으로 피할 수 없었습니다.

**해결**

이 한계를 확인한 뒤 **필요한 데이터를 자체 DB에서 해결하는 Kafka 기반 이벤트 동기화 전략**으로 방향을 전환했습니다.

### 문제 2 — Auth 서비스 설계 (JWT + Redis 기반 토큰 관리)

단순 세션 방식 대신 Access Token(단기) + Refresh Token(장기) 이중 구조를 표준화했습니다.

**핵심 설계 결정**

| 결정 | 이유 |
|------|------|
| tokenType 클레임 명시 | Refresh Token을 API 호출에 직접 사용하는 오남용 방지 |
| Refresh Token → Redis 저장 | Stateless JWT의 단점(서버 강제 만료 불가)을 Redis로 보완 |
| RTR(Refresh Token Rotation) | 기존 Refresh Token 폐기 후 신규 교체로 재사용 공격 차단 |
| 내부 전달 헤더 주입 | 하위 서비스에서 중복 인증 로직 없이 인가 처리 가능 |

**결과**
- ✅ 토큰 역할 분리로 보안성과 UX 동시 확보
- ✅ 다중 인스턴스 환경에서도 Redis 중앙 저장소로 인증 상태 일관성 확보
- ✅ 인증 책임 중앙화로 도메인 서비스 비즈니스 집중도 향상

---

### 문제 3 — EMS 서비스: 이벤트 기반 동기화 + 권한 자동 등록

**Outbox + Kafka Connect CDC 기반 동기화**

직원/조직/부서 변경 시점에 `OutBoxEvent`를 같은 트랜잭션 안에서 기록하고, Kafka Connect CDC가 Outbox를 읽어 타 서비스로 전달하도록 설계해 애플리케이션 코드에서 직접 전송 책임을 분리했습니다.

이벤트 타입은 `CREATE / UPDATE / DELETE / CREATE_BATCH / RESIGN / RESIGN_REVERT`로 표준화해 운영 케이스를 도메인 행위로 흡수했습니다.

**권한 메타데이터 자동 등록**

컨트롤러 메서드에 `@ApiPermission`을 선언하면 애플리케이션 기동 시 스캐너가 경로/메서드/설명을 자동 수집해 `msa_api_permissions`를 서비스 단위로 전체 교체 저장하도록 구성했습니다. 배포 시 권한 누락 리스크를 줄이는 것이 목적이었습니다.

---

### 문제 4 — LMS 서비스: 이벤트 멱등성 보장

LMS는 EMS Outbox + Kafka Connect CDC 파이프라인에서 발행된 이벤트를 수신해 자체 DB에서 처리합니다. Kafka의 at-least-once 전달 특성상 동일 메시지가 재전송될 수 있기 때문에, 메시지 수신 시 `eventId`를 `InboxEvent` 테이블에서 중복 확인하는 방식으로 멱등성을 보장했습니다.

연차 도메인 설계에서는 `totalDays`, `usedDays` 같은 집계 필드를 엔티티 컬럼으로 관리하면 이력 변경 시마다 함께 업데이트해야 해서 정합성 관리가 복잡해지는 문제가 있었습니다. 이를 해소하기 위해 **집계 필드를 제거하고 이력 테이블에서 그때그때 계산하는 방식**으로 전환했습니다.

### 문제 5 — 공통 라이브러리 및 배포 자동화 표준화

**공통 라이브러리 (GitHub Packages 배포)**

MSA 구조에서 서비스가 완전히 분리되다 보니 인증 헤더 파싱, 예외 처리, 로깅 같은 공통 코드를 서비스마다 반복 작성해야 하는 문제가 생겼습니다. 공통 모듈을 GitHub Packages 기반 라이브러리로 추출해 각 서비스가 의존성으로 가져다 쓸 수 있도록 구성했습니다.

- 인증 컨텍스트 필터 (Gateway 헤더 파싱 → `EmployeeContext` 표준화)
- 요청/응답 로깅 (MDC 기반, 5xx → error / 4xx → warn 자동 분기, ELK 연동 대비)
- 예외 처리 표준 포맷

**배포 자동화**

## 🏗️ 핵심 설계 포인트

### 서비스 경계 재정의 원칙

기존 MES/ERP는 서비스 경계가 **부서 단위**로 구분되어 도메인이 복잡해질수록 여러 도메인의 관심사가 뒤섞이는 문제가 있었습니다.

이번 전환에서는 **"어느 부서가 사용하는가"가 아니라 "어떤 비즈니스 책임을 가지는가"** 를 기준으로 경계를 재정의했습니다.

| 서비스 | 책임 |
|--------|------|
| Project | 프로젝트와 아이템 생명주기 관리만 담당 |
| Production | 생산 계획 및 지시/실적 관리 |
| Inventory | 재고 입출고 및 현황 |
| Product | 제품 정보 (기존 Master에서 분리) |
| Quality | Production과 강하게 결합되어 있던 품질 검사를 독립 서비스로 재정의 |
| Purchase | 구매 요청/발주 관리 |

서비스 간 연관은 직접 참조 대신 이벤트 기반 또는 참조 ID로 느슨하게 연결합니다. 단, 복잡한 집계 쿼리가 필요한 경우 해당 서비스의 DB에 이벤트로 동기화해 서비스 독립성을 유지하면서도 쿼리 요구사항을 충족하는 방식을 적용했습니다.

## 💡 핵심 문제 해결

### 문제 1 — 대용량 외부 데이터 수집 파이프라인 (Spring Batch ETL)

**배경**

2번 프로젝트에서 `AtomicBoolean` 기반 스케줄 동기화로 구현했던 Vault(외부 PLM) 연동을 Spring Batch 기반 ETL 파이프라인으로 전환했습니다. 기존 구조의 한계는 명확했습니다.

- 실패 시 처음부터 재실행해야 하는 구조
- 트랜잭션 범위가 컨트롤러 레이어까지 과도하게 길어지는 문제
- 단일 인스턴스에서만 유효한 `AtomicBoolean` 중복 방지

**해결**

Vault는 시간 단위로 전체 데이터를 제공하기 때문에, 매번 전체를 MySQL에 적재하면 중복 처리 비용이 커집니다. 이를 해소하기 위해 **MongoDB를 변경 감지 레이어로 활용**했습니다.

```
Vault → MongoDB 1차 적재 → 기존 데이터와 비교 → 변경분만 MySQL 이관
```

폴링 주기를 줄이거나 수동 트리거를 해도 MongoDB 비교 단계를 거치기 때문에 MySQL 쪽 부하는 일정하게 유지됩니다. Chunk 기반 처리로 트랜잭션 범위를 줄이고 실패 지점부터 재처리할 수 있는 구조도 확보했습니다.

**결과**
- ✅ MongoDB 변경 감지 레이어로 중복 처리 비용 최소화
- ✅ 수동 트리거 / 스케줄 폴링 모두 동일한 파이프라인으로 처리
- ✅ 배치 실패 시 부분 재처리 가능한 구조로 데이터 유실 리스크 감소

### 문제 2 — 동기/비동기 통신 설계 기준

서비스 간 통신에서 동기 API 호출과 비동기 이벤트를 무조건 하나로 통일하지 않고, **비즈니스 정합성 보장 필요 여부**를 기준으로 명확히 구분해 설계했습니다.

- **동기 API**: 재고 차감/반영, 생산 실적 상태 변경처럼 데이터 정합성이 핵심인 흐름
- **비동기 이벤트**: 알림/푸시, 통계 집계, 이력 적재처럼 즉각적인 정합성보다 처리 완료 시점이 유연해도 되는 부가 처리
