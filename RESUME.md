### WSA MES/ERP 개발

2024.03 ~ 2024.12

역할: MES(생산관리시스템)에 필요한 도메인 설계 및 멀티 모듈 서비스 별(영업,생산,재고,구매) 기능 개발

#### 프로젝트 개요

기존 단순 CRUD 기반 프로젝트 관리를 대체하여, 수주부터 설계·생산·출하까지 전반적인 업무 흐름을 관리하는 ERP/MES를 설계·개발하였습니다.
이후 운영 단계에서 신규 기능 개발과 레거시 구조 개선을 병행하였습니다.

#### 담당 업무

- 영업, 생산, 재고, 구매 멀티 모듈의 도메인 설계
  - 실무 기획과 개발을 직접 주도하며 각 팀과 요구사항 미팅을 반복적으로 진행하여 업무 프로세스를 파악하고 모듈별 서비스 경계를 정의하였습니다.
  - MSA 기반 아키텍처를 초기에 검토하였으나, 개발 속도와 데이터 정합성을 우선하여 단일 DB 기반에 애플리케이션 레벨 모듈 분리 구조를 채택하였습니다.

- 생산 관리 ERP 시스템의 프로젝트, 품목, 공정, 생산 흐름 기능 구현
  - 프로젝트를 통해 수주된 품목을 생산 관리하고, 해당 품목에 대한 생산지시 기능을 구현하였습니다.
  - 제품-하위부품 관계를 자기 참조 트리 구조로 설계하여 다단계 BOM을 표현하였고, 공정 템플릿 기반 라우팅과 생산 BOM 인스턴스로 유연한 작업 흐름을 구현하였습니다.

- 구매 관리 ERP 시스템의 구매 요청 및 발주 흐름 기능 구현
  - BOM 기준으로 구매 자재와 필요 수량을 산출하는 QueryDSL 기반 조회 기능을 구축하고, 조회 결과를 기반으로 구매 요청 및 발주 흐름을 설계하였습니다.
  - 발주된 자재의 입출고·재고 반영 시 구매 모듈과 재고 모듈 간 내부 서비스를 동기적으로 연계하였고, 예외 발생 시 Saga 패턴으로 데이터 정합성과 원자성을 보장하였습니다.

- 재고 관리 ERP 시스템의 재고 처리 기능 구현
  - 생산·구매·영업 등 여러 모듈에서 공통으로 사용하는 내부 서비스로 설계하였으며, 각 서비스의 실적과 연계하여 입출고 이력을 추적할 수 있는 구조를 구현하였습니다.
  - 입출고 원인 데이터를 공통 추상 구조로 설계하여, 각 서비스가 실적을 생성한 뒤 재고 서비스를 호출하면 해당 실적을 참조해 재고가 처리되도록 하였습니다.
  - 동시 처리 시 음수 재고가 발생할 수 있는 문제를 확인하고, 검증과 업데이트를 단일 쿼리로 처리하도록 개선하였습니다.

- 레거시 구조 개선
  - 기존 사내 공통 라이브러리(BaseService, BaseController, QueryParameter 등)는 단순 CRUD 중심으로 설계되어 복잡한 비즈니스 요구사항 반영에 한계가 있었고, 일부 도메인에서는 적합하지 않음에도 관성적으로 적용되고 있었습니다.
  - QueryParameter 기반 Specification/DB Path Builder 방식은 복합 조건·계산 로직·동적 검색 처리에 한계를 드러냈고, 애플리케이션 레벨 보조 로직으로 보완하였으나 유지보수 부담이 커졌습니다.
  - 조회 로직을 QueryDSL 기반으로 전환하여 복잡한 동적 조회를 유연하게 처리할 수 있도록 개선하였습니다.


### WSA MSA 인프라 개발

2025.02 ~ 2025.09
역할: MSA 전환 아키텍처 설계, 공통 서비스 개발, 공통 인프라 개발, 파일럿 서비스 구축

#### 프로젝트 개요

MES/ERP 운영 중 단일 DB 기반의 데이터 결합도 증가·확장성 한계를 경험하여 MSA 전환을 추진하였습니다.
즉각적인 전체 전환보다 공통 인프라와 공통 서비스를 먼저 구축하고, 직원 관리·인증 서비스를 선행 개발한 뒤 연차 관리 시스템을 파일럿 서비스로 구현하여 구조를 검증하였습니다.

#### 담당 업무

- 서비스 간 데이터 동기화 구조 설계
  - 서비스 분리가 진행될수록 데이터의 분산·중복이 불가피함을 확인하여, 데이터 동기화를 전제로 한 구조 설계로 방향을 전환하였습니다.
  - Redis 기반 이벤트 동기화를 직접 구현하였으나 다음과 같은 한계를 확인:
    - 이벤트 처리 로직이 서비스별로 반복 구현되며 코드 복잡도 급격히 증가
    - 이벤트 유형별 분기 처리로 유지보수 비용 증가
    - 신규 서비스 추가 시 생산자·소비자 양쪽 모두 구현이 필요하여 확장성 저하
  - Redis Pub/Sub도 검토하였으나 메시지 유실 가능성과 구조적 한계로 Kafka를 도입하여 재설계:
    - Outbox 패턴으로 트랜잭션과 이벤트 발행의 원자성 보장
    - eventId·action 기반 설계로 소비자가 독립적으로 처리 가능
    - Inbox 패턴으로 중복 이벤트 처리 방지 (idempotency 확보)
  - 이벤트 방식 외에 Spring Batch 기반 전체 데이터 동기화 구조도 설계하였으며, Chunk 100 단위로 API에서 읽어 가공·저장하는 방식으로 추후 MSA 전환 시 DB 이관에 활용할 수 있도록 구조화하였습니다.

- 테스트 코드 구조 표준화
  - 팀 내 테스트 작성 기준이 없어 개인별로 제각각이었기에, MSA 인프라 설계 시 표준 구조를 함께 정립하였습니다.
  - Mock 단위 테스트보다 DB·Kafka 등 실제 환경을 검증하는 통합 테스트를 우선하여 TestContainers 기반 구조로 표준을 잡았으며, 서비스별 추가 설정은 TestConfig로 확장하도록 하였습니다.
  - 각 API는 최소 성공 1개, 실패 1개 케이스 이상 작성을 기준으로 정하였으며, 유효성 검사는 Controller 단 `@Valid`와 Controller 테스트에서 함께 검증하도록 구성하였습니다.

- 파일럿 서비스를 통한 MSA 시스템 검증
  - 기존 엑셀로만 처리되던 사내 연차 관리를 파일럿 서비스로 선정하여 구축하였습니다.
  - 연도별 휴가 정보에 발생·사용 기록을 반영하는 방식으로 도메인을 설계하였습니다.
  - 직원 관리·인증 공통 서비스와 연계하여 서비스 분리 구조와 운영 방식을 실제로 검증하였습니다.
  - Kafka 기반 직원·부서 정보 동기화가 정상 동작하는 것을 파일럿 서비스를 통해 검증하였습니다.

- GitHub Actions를 통한 배포 자동화
  - Gitea/Jenkins 기반 프로젝트별 분산 구성을 GitHub Actions 기반으로 일괄 모듈화하였습니다.
  - QA: Push 시 자동 덮어쓰기 / Deploy: Blue/Green 방식 무중단 배포로 환경을 분리하여 구성하였습니다.

- MSA 공통 인프라 라이브러리 설계 및 개발 (엔티티, 예외 처리, 로깅, DTO 등)
  - 각 서비스 공통 코드를 라이브러리로 추출하되, 상속·인터페이스·오버라이딩으로 서비스별 유연한 구성이 가능하도록 설계하였습니다.
