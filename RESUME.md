### WSA MES/ERP 개발

2024.03 ~ 2024.12

역할: 도메인 설계 및 멀티 모듈 서비스 별(영업,생산,재고,구매) 기능 개발

#### 프로젝트 개요

기존 회사에서 있던 프로젝트 관리 서비스가 있었으나 너무 기본적인 CRUD만 관리하게끔 되어 있었기에 스마트팩토리 구현을 위해
프로젝트 수주부터 설계, 생산, 출하까지 이어지는 전반적인 업무 흐름을 관리하는 ERP 및 MES(생산관리시스템)를 설계 및 개발 하였습니다.
이후 운영 단계에서 신규 기능을 지속적으로 개발하는 동시에, 레거시 구조를 점진적으로 개선하였습니다.

#### 담당 업무

- 영업, 생산, 재고, 구매 멀티 모듈의 도메인 설계                                                                                                                                                     
  - 당시 실무 기획과 개발을 사실상 직접 주도해야 하는 상황이었기 때문에, 회사의 업무 프로세스와 생산 관련 운영 흐름을 먼저 파악하고 이를 시스템 설계로                                 
  연결해야 했습니다.                                                                                                                                                                                    
  - 각 팀과 반복적으로 요구사항 미팅을 진행하며 업무 프로세스를 분석하고, 모듈별 서비스 경계를 정의하였습니다.                                                                                         
  - 초기에는 MSA 기반 아키텍처를 검토하였으나, 빠른 개발 속도와 데이터 정합성 확보가 더 중요하다고 판단하여 우선 단일 DB 기반으로 구성하고 애플리케이션 레벨에서 모듈을                                                
  분리하는 구조를 채택하였습니다.                                                                                                                                                                               
  - 이를 기반으로 각 모듈의 도메인 구조와 데이터 흐름을 설계하였으며, 이후에도 실무 피드백을 반영해 도메인 모델, 엔티티 관계, 업무 흐름을 지속적으로 보완하고                                          
  구체화하였습니다.                                                                                                                                                           

  - 생산 관리 ERP 시스템의 프로젝트, 품목, 공정, 생산 흐름 기능 구현                                                                                                            
    - 프로젝트를 통해 수주된 품목을 생산 관리하고, 해당 품목에 대한 생산지시 기능을 구현하였습니다.                                                                                                 
    - 이 과정에서 제품-하위부품 관계를 자기 참조 트리 구조로 도메인 설계하여 다단계 BOM 및 생산 계층 구조를 표현하였고, 공정 템플릿 기반의 라우팅과 생산 BOM 인스턴스를                                                  
      활용해 실제 생산 과정에서 유연하게 작업 흐름을 관리할 수 있도록 구현하였습니다.

  - 구매 관리 ERP 시스템의 구매 요청 및 발주 흐름 기능 구현
    - 프로젝트별 품목에 필요한 구매 자재를 조회한 뒤 발주로 이어지는 업무 흐름을 설계하고 구현하였습니다.
    - 먼저 정확한 구매 대상과 수량을 산출하는 것이 중요하다고 판단하여, 프로젝트에 포함된 품목들을 BOM 기준으로 분석하고 구매품으로 분류된 자재와 필요 수량을 조회할 수                                                  
      있는 Querydsl 기반 기능을 구축하였습니다.
    - 이후 사용자가 조회된 자재를 기준으로 발주를 진행할 수 있도록 구매 요청 및 발주 흐름을 설계하였습니다.
    - 또한 발주된 자재의 입출고 및 재고 반영 과정에서 구매 관리 모듈과 재고 관리 모듈 간 내부 서비스를 동기적으로 연계하였고, 예외 발생 시에는 Saga 패턴을 적용해 데이터                                                 
      정합성과 원자성을 보장하였습니다.

  - 재고 관리 ERP 시스템의 재고 처리 기능 구현
    - 재고 모듈은 생산, 구매, 영업 등 여러 서비스에서 공통적으로 사용하는 내부 서비스 역할을 수행해야 했기 때문에, 각 도메인에서 일관되게 활용할 수 있는 공통 재고 처리                                                  
      구조를 만드는 데 중점을 두었습니다.
    - 또한 입고 및 출고 내역이 어떤 업무 사유와 실적에 의해 발생했는지 추적할 수 있는 구조가 필요하다는 요구사항을 반영하였습니다.
    - 이를 위해 각 서비스에서 발생하는 입출고 원인 데이터를 공통 추상 구조로 설계하고, 각 서비스가 실적 정보를 생성한 뒤 재고 서비스의 입출고 기능을 호출하면 해당 실적을                                                
      참조하여 재고가 처리되도록 구조를 설계하였습니다.
    - 테스트 과정에서는 동시성으로 인한 덮어쓰기와 동시 처리 시 음수 재고가 발생할 수 있는 문제를 확인하였고, 검증과 업데이트를 단일 쿼리로 처리하도록 개선하여 이를                                                     
      방지하였습니다.
    - 추가로 재고 이력을 기반으로 입출고 내역을 재정비할 수 있는 리프레시 기능을 구현하였고, 각 서비스의 실적 고유값을 활용해 이력 정합성도 함께 높였습니다.

  - 레거시 구조 개선
    - 기존 사내 공통 라이브러리인 BaseService, BaseController, QueryParameter 등은 여러 도메인과 조회 API 전반에 공통적으로 사용되고 있었으나, 단순 CRUD 중심으로 설계되어                                               
      복잡한 비즈니스 요구사항을 반영하는 데 한계가 있었습니다.
    - 특히 일부 도메인에서는 해당 구조가 실질적으로 적합하지 않음에도 관성적으로 적용되고 있었고, 이로 인해 도메인 특화 로직과 사용자 관점의 업무 흐름이 코드에 자연스럽게                                               
      반영되지 못하는 문제를 확인하였습니다.
    - QueryParameter 기반의 Specification/DB Path Builder 방식은 데이터베이스 필드 조건 중심의 단순 조회에는 적합했지만, 생산관리 시스템에서 요구되는 복합 조건, 계산 로직,                                             
      동적 검색 시나리오를 처리하는 데 한계를 드러냈습니다.                                                                                                                    
    - 초기에는 애플리케이션 레벨의 보조 계산 로직으로 이를 보완하였으나 유지보수성과 확장성 측면에서 제약이 있다고 판단하였고, 이후 조회 로직을 Querydsl 기반으로 전환하여  복잡한 동적 조회를 유연하게 구현 할 수 있도록 개선하였습니다.


### WSA MSA 인프라 전환

2025.02 ~ 2025.09
역할: 아키텍처 설계 및 구축 , 공통 인프라 개발 , 파일럿 서비스를 통한 MSA 환경 구축

#### 프로젝트 개요

회사 진행 했던 MES/ERP 프로젝트의 단일 DB의 한계를 느껴 MSA로 전환 프로젝트를 하였습니다.
다만 그 이전에 인프라적으로 준비 해야 할 것들이 많아 먼저 필요한 서비스들을 구축하기로 진행하였습니다.
구축 할것은 아키텍처 전체적인 설계와 공통적으로 사용 할 직원 관리 서비스와 로그인 서비스 , 해당 기능들을 먼저 테스트 해볼 파일럿 테스트로 자사 연차 관리 서비스를 진행하기로 하였습니다.

#### 담당 업무

- MSA 아키텍처 구조 설계
  - 가장 먼저 필요한 서비스들을 정의 하고 서비스들의 경계를 짓는 것이 필요하였습니다.
  - 이후는 MSA를 개발을 진행하면서 필요한 Gateway에 붙일 유레카도 있었지만 회사내에서 로봇 관련 시스템으로 C#를 사용하고 있었던 부분도 고려하여 더 다양한 프레임워크와 툴에서 사용되는 Consul 서버를 사용하였습니다.
- 서비스 간 데이터 동기화 구조 설계
  - 처음에는 최대한 중복 DB가 없게끔 진행하고 싶었으나 서비스 구축에 따라서 관점에 따라서는 테이블처럼 중복이 될 수 있는 상황이 있다고 판단하였습니다.
  - 
- Gateway 전역 인증 필터 및 동적 라우팅 구성
- 배포 자동화 및 개발 환경 표준화
