### WSA MES/ERP 개발

2024.03 ~ 2024.12

역할: 도메인 설계 및 멀티 모듈 서비스 별(영업,생산,재고,구매) 기능 개발

#### 프로젝트 개요

기존 회사에서 있던 프로젝트 관리 서비스가 있었으나 너무 기본적인 CRUD만 관리하게끔 되어 있었기에 스마트팩토리 구현을 위해
프로젝트 수주부터 설계, 생산, 출하까지 이어지는 전반적인 업무 흐름을 관리하는 ERP 및 MES(생산관리시스템)를 설계 및 개발 하였습니다.
이후 운영 단계에서 신규 기능을 지속적으로 개발하는 동시에, 레거시 구조를 점진적으로 개선하였습니다.

- 영업, 생산, 재고, 구매 멀티 모듈의 도메인 설계

  연결해야 했습니다.                                                                                                                                                                                    
  - 각 팀과 반복적으로 요구사항 미팅을 진행하며 업무 프로세스를 분석하고, 모듈별 서비스 경계를 정의하였습니다.                                                                                         
  - 초기에는 MSA 기반 아키텍처를 검토하였으나, 빠른 개발 속도와 데이터 정합성 확보가 더 중요하다고 판단하여 우선 단일 DB 기반으로 구성하고 애플리케이션 레벨에서 모듈을

  구체화하였습니다.                                                                                                                                                           

- 생산 관리 ERP 시스템의 프로젝트, 품목, 공정, 생산 흐름 기능 구현                                                                                                            
  - 프로젝트를 통해 수주된 품목을 생산 관리하고, 해당 품목에 대한 생산지시 기능을 구현하였습니다.

### WSA MSA 인프라 전환

2025.02 ~ 2025.09
역할: 아키텍처 설계 및 구축 , 공통 인프라 개발 , 파일럿 서비스를 통한 MSA 환경 구축

#### 프로젝트 개요

기존 MES/ERP 프로젝트를 진행하며 단일 DB 기반 구조에서 데이터 결합도 증가, 트랜잭션 복잡성, 확장성 한계 등의 문제를 경험하였습니다.
이에 따라 서비스 간 독립성과 확장성을 확보하기 위해 MSA 전환을 추진하였습니다. 다만, 즉각적인 전환보다는 안정적인 아키텍처 구축을 우선으로 판단하여,
MSA 전환에 필요한 공통 인프라와 핵심 서비스를 선행 구축하는 방향으로 진행하였습니다. 먼저 전체 아키텍처를 설계하고, 모든 서비스에서 공통으로 활용할 수 있는
직원 관리 서비스와 인증/로그인 서비스를 분리하여 구축하였습니다. 이후, 해당 구조의 안정성과 운영 가능성을 검증하기 위해
사내 연차 관리 시스템을 파일럿 서비스로 선정하여 적용 및 테스트를 진행하기로 하였습니다.

#### 담당 업무

- 서비스 간 데이터 동기화 구조 설계

초기에는 서비스 간 데이터 중복을 최소화하기 위해 단일 DB에 가까운 구조를 지향했으나,
서비스 분리가 진행될수록 각 서비스의 책임과 관점에 따라 데이터가 분산·중복될 수밖에 없음을 확인하였습니다.

이에 따라 데이터 동기화를 전제로 한 구조 설계로 방향을 전환하였습니다.

초기에는 Redis를 활용하여 이벤트 기반 동기화 구조를 직접 구현하였습니다.
데이터 변경 시 payload를 Redis에 저장하고, 내부 서비스로 비동기 요청을 전파하는 방식으로 구성하였으며,
기본적인 재시도 로직도 함께 적용하였습니다.

다만, 다음과 같은 한계를 확인하였습니다:

이벤트 처리 로직이 서비스별로 반복 구현되며 코드 복잡도가 급격히 증가
Create/Update/Patch 등 이벤트 유형별 분기 처리로 유지보수 비용 증가
신규 서비스 추가 시 생산자·소비자 양쪽 모두에 구현이 필요하여 확장성이 저하
비동기 처리 특성상 예외 및 정합성 관리에 대한 부담 증가

이러한 문제를 해결하기 위해 Redis Pub/Sub도 검토하였으나,
메시지 유실 가능성과 구조적 한계로 인해 보다 안정적인 메시징 시스템이 필요하다고 판단하였습니다.

최종적으로 Kafka를 도입하여 이벤트 기반 데이터 동기화 구조를 재설계하였습니다.

Outbox 패턴을 적용하여 트랜잭션과 이벤트 발행의 원자성을 보장
이벤트에 eventId와 action을 정의하여 소비자가 독립적으로 처리할 수 있도록 설계
Inbox 패턴을 적용하여 중복 이벤트 처리 방지 (idempotency 확보)
서비스 간 직접 호출 없이 메시지 기반으로 데이터 동기화 수행

이를 통해 서비스 간 결합도를 낮추고,
확장성과 데이터 정합성을 동시에 확보하는 구조로 개선하였습니다.

추후에는 이벤트 만이 아니라 전체적인 동기화도 해주는 것도 설계가 필요하여 Spring Batch 전체 대용량 데이터를 동기화 하는 구조도 설계 하였습니다.
지금은 부서나 직원 정보정도라서 작은 데이터 정도라서 안맞는다근ㄴ 판단도 하였으나 추후 MSA 전환 할때 추후 서비스들의 DB데이터를 옮겨주는 작업이나 
동기화 패턴을 일정하게 구조화 하는데에도 많은 도움이 될 것이라는 판단이었습니다. 

Chunk 100을 단위로 하여 Read할때 API를 통해서 데이터 정보를 가져오고 이후 맞는 엔티티에 가공 후에 저장하는식으로 작업을 진행하였습니다.

- 테스트 코드 구조 표준화

기존 회사에서 테스트 코드를 제대로 구조화를 하지 않고 개인이 알아서 하는식으로 구조였고 이것조차도 사람마다 편차가 심했기에 MSA 쪽 공통 인프라 설계하면서 만져봤습니다.
기본적으로 Mock을 이용한 서비스 로직보다는 통합적으로 DB에 데이터가 올바르게 생성 혹은 값이 설정되었는지 확인하는것이 가장 중요하다고 생각했기에 일단 TestContainer를 활용한 
테스트로 구조로 잡았습니다. DB뿐만 아니라 Kafka 같은 환경도 테스팅이 가능했고 일부 Config 작성이 필요한 경우 각 서비스마다 TestConfig를 작성하면 되는식으로 진행하였습니다.
기본적으로 각 API는 1차적으로 성공케이스 하나와 실패케이스 하나 총 2개이상의 최소 케이스를 만들고 Vaild 의 경우 기본적으로 Controller 단에 유효성 @Vaild로 유효성 검사를 하고 있기에 이것을 Controller 테스트에서 제대로 케이스를 확인하는식으로 하였습니다.


- 파일럿 서비스로 인한 MSA 서비스 시스템 테스팅
파일럿 서비스로 연차 관리 서비스를 구축하였습니다. 기존 회사에서 엑셀로만 처리 하였고 전산상으로 각 짃원들이 연차 관련 정보를 처리하는 시스템이 없었기에 이번에 파일럿 서비스로 적합하다는 생각을 하였습니다.
연도 별 휴가 정보에 발생 기록들과 사용 기록들을 통해 해당 휴가 정보를 업데이트 하는식으로 도메인 구조를 잡았고 해당 서비스는 Kafka 서비스에 있는 직원 정보 ,부서 정보 동기화를 통해 훌륭히 데이터 동기화가 잘 되는 것을 확인화였습니다.


- GitHub Action 을 통한 배포
기존에는 Git tea라는 것을 통해 이용하였고 해당 웹훅으로 몇몇 서비스들은 젠킨스를 통해 배포 작업을 하였지만 각 프로젝트마다 달라지는 웹훅보다는 이번에 깃허브로 변경하게 되면 깃허브액션을 통해 일괄적으로 모듀화 작업을 진행하였습니다.
QA , Deploy 환경 모두 다르게 적용되도록 진행하였고 QA 환경에서는 Push 되면 자동적으로 서버가 덮어씌어지게끔 Deploy 환경에서는 Blue, green 식으로 인스턴스를 2개를 순간적으로 띄우고 하나로 교체하는식으로 진행하였습니다.
특정 서비스의 경우는 다중 인스턴스가 될 서비스가 존재 할 수 있기에 이것도 확장 될 것을 감안하고 다른 깃허브 모듈화를 고려중입니다.

- MSA Infra LIb 개발
MSA 특성상 각 서비스가 공통적으로 가져야 할 코드들이 많아 이것들을 공통화 하는 라이브러리를 작성했습니다.
일단 BaseEntity라는 최소 기본적으로 엔티티를 abstrarct 처리 하였고 elk 에서 logstash로 보내주는 기본적으로 ReqeustLoogger, ResponseLoogger를 만들고 각 서비스에서 Bean으로 수동으로 등록해서 사용하는 방식을 채택하였습니다.
각 서비스마다 공통적으로 처리 할 수 있지만 다르게 처리 할 수도 있기에 상속을 통해 받거나 인터페이스로 활용해서 혹은 오버라이딩을 할 수 있게끔 할 수 있는것을 고렿 ㅐㅆ습니다. 
해당 기능은 익셉션 , 로그 , 엔티티 및 DTO 관련 공통 처리를 하였습니다. Kafka나 뭐 다른 DB Config작업은 각 서비스단에서 자유도를 주고 싶었기에 공통화로 묶지 않았고 Kafka 코드를 작성할때 패턴등은 인터페이스나 코드로 뺴는것이 좋을것 같아 구조화 하여 처리 할 수 있게끔 하였습니다.
