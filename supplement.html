김종훈 — 추가 질문 답변

Q1

다양한 기술을 어떻게 활용하여 어떤 문제를 해결했는지

MSA 전환 과정에서 서비스 간 데이터 동기화 구조는 세 단계를 거쳐 변화했습니다. 각 단계마다 문제를 직접 경험하고 나서야 다음 방향이 명확해졌습니다.

  • 1
    Redis 저장 + REST 클라이언트 비동기 호출 처음에는 단순하게 접근했습니다. 동기화가 필요한 데이터를 Redis에 저장해두고, 각 외부 서비스에 REST 클라이언트로 비동기 요청을 보내는 방식이었습니다. 빠르게 동작했지만, 도메인마다 그리고 도메인의 이벤트마다 다르게 비동기 요청을 구성해야 했고 각 소비자측 서비스에서도 해당 API 컨트롤러 단을 전부 구현해야 했기에 많은 코드가 발생하는 단점이 있었습니다.
  • 2
    Redis Pub/Sub으로 전환 — 그리고 구조적 한계

    기존의 직접 호출 방식은 서비스가 늘어날수록 호출 코드가 기하급수적으로 증가하는 문제가 있었습니다. 이를 해결하기 위해 서비스 간 결합도를 낮출 수 있는 Redis Pub/Sub 기반의 구독 방식으로 전환했습니다.

    그러나 Pub/Sub 구조의 한계가 드러났습니다. 도메인과 이벤트 종류가 증가할수록 채널 및 구독 대상이 함께 늘어나면서 관리 복잡도가 크게 증가했습니다. 이벤트 타입에 따라 단일 채널을 쓰면 소비자 분기 로직이 복잡해지고, 채널을 분리하면 채널 수가 급증하는 딜레마가 생겼습니다. 그 결과 소비자 서비스마다 유사한 분기 처리 및 메시지 파싱 로직이 반복적으로 구현되는 비효율이 발생했습니다.

    더불어 Redis Pub/Sub은 메시지를 저장하지 않는 구조이기 때문에, 구독자가 일시적으로 비활성화되거나 장애가 발생할 경우 메시지가 유실될 수 있었고, 서비스 간 데이터 정합성을 보장하기 어려웠습니다.

  • 3
    Kafka 전환 — 구조를 바꾸다

    Redis에 기능을 계속 보완하는 방식으로는 근본적인 해결이 어렵다고 판단하여 Kafka 기반 구조로 전환했습니다.

    Kafka는 이벤트를 로그 형태로 저장하기 때문에 메시지 유실 없이 재처리가 가능하며, 신규 서비스가 추가되더라도 과거 이벤트를 기반으로 상태를 재구성할 수 있습니다. Outbox 패턴으로 비즈니스 트랜잭션과 이벤트 발행의 원자성을 확보하고, Inbox 패턴으로 중복 소비를 제어하여 소비자 측 데이터 정합성을 보장했습니다.

    이벤트 발행은 CDC 기반의 Kafka Connect로 이관하여 서비스 코드에서 발행 로직을 분리했으며, 소비자는 토픽과 메시지 헤더의 이벤트 타입을 기준으로 필요한 이벤트만 선택적으로 처리하도록 구조를 단순화했습니다. 초기 데이터 이관이나 수동 재동기화가 필요한 상황을 위해 관리자 API를 통한 전체 동기화 기능도 함께 구현했습니다.

Q2

가장 임팩트 있었던 경험, 또는 큰 실패를 극복한 과정

MES/ERP — end-to-end 경험에서 배운 것들

도메인 설계부터 개발·배포·운영까지 참여한 프로젝트였습니다. 개발 이후 배포 및 운영을 하면서 많은 것을 배웠습니다.

  • 1
    운영 단계 요구사항 검증 — 정답이 없는 요구사항을 판단하는 과정 운영 단계에서는 다양한 사용자 요구사항이 지속적으로 발생하며, 이를 단순히 구현하는 것이 아니라 적절히 판단하는 과정이 필요했습니다. 초기에는 일부 데이터를 개발자가 쿼리로 처리하면 충분하다고 판단했지만, 실제로는 사용자가 직접 조회하고 관리할 수 있는 기능이 요구되었습니다. 이를 통해 사용자 관점에서의 필요를 놓칠 수 있음을 인지했습니다. 반대로 결재 단계를 세분화해달라는 요청은 시스템 복잡도와 업무 효율 저하를 고려하여 제한적으로 반영했습니다. 이러한 경험을 통해 요구사항은 단순 구현의 대상이 아니라, 사용자 가치와 시스템 구조 사이에서 균형 있게 판단해야 하는 설계의 일부임을 체감했습니다.
  • 2
    코드 관리 — 개발할 때는 몰랐던 무게

    개발 중에는 코드 관리 방식에 큰 신경을 쓰지 않았습니다. 기능 완성과 로직 구현에만 초점을 두었기 때문입니다. 그런데 운영 단계에 들어서면서 상황이 달라졌습니다. 코드를 수정하거나 리팩토링하는 과정 자체에 예상보다 훨씬 많은 리소스가 들었습니다. 직관성이 떨어지는 코드는 의도를 파악하는 데만 오랜 시간이 걸렸고, 통일성이 없는 코드는 팀원마다 작성 방식이 달라 차이점을 비교하고 이해하는 데 또 시간이 빠졌습니다.

    명확한 기준이 없으니 코드 리뷰에서 팀원들에게 근거 있는 피드백을 주기가 어려웠고, 기준이 부재한 상태에서 누구나 자유롭게 코드를 푸시하고 배포할 수 있는 환경은 코드 품질을 더욱 빠르게 악화시켰습니다. 코드는 작성하는 것보다 유지하는 것이 더 어렵고, 처음부터 관리 가능한 구조와 팀 차원의 컨벤션을 갖추는 것이 중요하다는 것을 이때 깨달았습니다.

  • 3
    아키텍처 선택의 트레이드오프 — 나중에야 보이는 비용

    MES/ERP 초기 설계 시 MSA와 단일 DB + 애플리케이션 레벨 모듈 분리 사이에서 고민했습니다. 당시에는 개발 속도와 생산 관리 시스템의 특성상 데이터 정합성을 우선했고, 단일 DB 기반의 모듈 분리 구조를 선택했습니다. 틀린 판단이었다고 생각하지는 않습니다. 그 시점에서는 합리적인 결정이었습니다.

    하지만 운영하면서 단일 DB의 한계가 점점 드러나기 시작했습니다. 데이터와 사용자 추가 요청사항이 늘어날수록 확장에 제약이 생겼고, 스키마를 변경하거나 마이그레이션을 할 때마다 전체 모듈에 영향을 줄 수 있어서 신경 써야 할 것이 많아졌습니다. 이 경험이 결국 MSA 전환을 직접 추진하게 된 계기가 되었습니다.

이 경험들이 가장 임팩트 있었던 이유는, 기술적인 성장보다 운영이라는 새로운 관점을 얻었기 때문입니다. 개발할 때는 "어떻게 만들 것인가"만 생각했다면, 운영 이후에는 "사용자가 어떻게 쓸 것인가", "문제가 생겼을 때 어떻게 대응할 것인가", "이 구조가 6개월 뒤에도 유효한가"를 함께 고려하게 되었습니다. 결정 당시엔 합리적이었던 선택이 시간이 지나면서 비용이 되는 것을 직접 경험하고 나서야, 아키텍처 결정이 왜 신중해야 하는지를 이론이 아닌 체감으로 알게 되었습니다.
Q3

오픈소스 라이브러리 기여 경험 또는 코드 분석 후 적용 경험

직접적인 오픈소스 기여 이력은 아직 없습니다. 다만 라이브러리의 내부 동작을 분석한 뒤 도입하거나, 기존 라이브러리의 동작 방식에서 영감을 얻어 사내 공통 라이브러리를 설계한 경험이 있습니다.

① TestContainers 도입

MSA 전환 과정에서 테스트 작성 기준이 부재하여, 서비스마다 테스트 방식과 실행 환경이 달라 통합 테스트의 일관성과 재현성이 떨어지는 문제가 있었습니다. 이를 해결하기 위해 Testcontainers의 공식 문서와 GitHub 소스를 분석한 뒤, 기존 라이브러리를 포크하여 사내 테스트 환경에 맞게 컨테이너 생명주기와 공유 방식을 확장했습니다. 이후 공통 컨테이너 설정을 추상 기반 클래스로 공통화하고, 여러 테스트 클래스가 동일한 환경을 손쉽게 재사용할 수 있도록 `@SharedContainers` 구조를 확정했습니다. 그 결과 테스트 환경 구성 코드의 중복을 줄이고, 서비스 간 통합 테스트 환경을 일관되게 유지할 수 있었으며, 컨테이너 기동 비용까지 절감하여 테스트 작성 및 실행 효율을 높였습니다. 다만 오픈소스 라이브러리의 버전 업데이트에 따라 내부 구현 변화에 지속적으로 대응해야 하는 유지보수 비용이 크다고 판단하여, 최종적으로는 공식 기능 기반으로 테스트 구조를 단순화하는 방향으로 개선했습니다.

② MSA 공통 인프라 라이브러리 설계

각 마이크로서비스가 공통으로 사용하는 예외 처리, 로깅, 기반 엔티티, DTO 구조를 사내 라이브러리로 추출했습니다. Spring의 @ControllerAdvice나 Spring Boot Auto-configuration의 확장 지점 설계를 참고하여, 서비스마다 상속·인터페이스·오버라이딩으로 유연하게 구성할 수 있도록 추상화 수준을 조율했습니다. "너무 강하게 추상화하면 서비스별 변형이 어렵고, 너무 느슨하면 공통화 의미가 없다"는 균형을 맞추는 과정이 프레임워크 내부 설계 철학을 실제로 이해하는 계기가 되었습니다.

Q4

팀의 생산성을 저하시키는 문제를 발굴하고 개선한 경험

두 가지 문제를 직접 발굴하여 개선했습니다. 하나는 배포 프로세스, 다른 하나는 테스트 문화입니다.

① 배포 자동화 — 분산된 파이프라인을 단일 구조로 통합

MSA 전환 이전에는 각 서비스가 Gitea + Jenkins 조합으로 개별 배포 파이프라인을 운영하고 있었습니다. 서비스마다 설정이 달라 배포 방식 변경이나 신규 서비스 추가 시 반복적인 수정이 필요했고, 배포 방식의 일관성과 추적성도 부족한 상태였습니다.

  • 1
    문제 발굴 — 배포 프로세스 중복과 관리 비용 증가 Github로 플랫폼을 옮겨서 GitHub Actions 도입 이후에도 각 서비스가 개별 워크플로우를 구성하면서 동일한 배포 로직이 반복적으로 작성되었습니다. 이로 인해 배포 방식 변경 시 여러 저장소의 설정을 동시에 수정해야 했고, 일부 서비스에만 변경이 반영되거나 설정이 누락되는 등 배포 일관성이 깨질 수 있는 구조적 리스크가 발생했습니다.
  • 2
    개선 — 재사용 가능한 배포 템플릿 설계 공통 배포 로직을 GitHub Actions의 reusable workflow(workflow_call)로 분리하고, 각 서비스에서는 해당 워크플로우를 호출하는 구조로 설계했습니다. QA 환경은 Push 시 자동 배포되도록 구성하고, 운영 환경은 무중단 배포와 빠른 롤백을 위해 Blue/Green 배포 방식을 적용했습니다. 서비스별로는 서비스명, 이미지 이름, 포트 등 최소한의 설정 값만 전달하도록 하여 배포 프로세스를 표준화했습니다.
  • 3
    결과 — 배포 일관성 확보 및 운영 안정성 향상 배포 방식 변경 시 중앙 워크플로우만 수정하면 전체 서비스에 일괄 반영이 가능해졌고, 설정 누락 없이 일관된 배포 프로세스를 유지할 수 있게 되었습니다. 또한 신규 서비스 추가 시 별도 파이프라인 구성 없이 최소 설정만으로 배포가 가능해져 온보딩 속도를 개선했습니다.

② 테스트 표준화 — "어떻게 테스트해야 하는가"에 대한 기준 수립

MSA 인프라 구축 초기에는 테스트 코드 작성 기준이 없어, 테스트 작성 여부와 방식이 개인마다 달라 일관성이 부족했습니다. 이를 해결하기 위해 테스트 전략을 명확히 정의했습니다. Mock 기반 단위 테스트보다 실제 DB와 Kafka를 사용하는 통합 테스트를 우선하도록 방향을 설정했습니다. 또한 "각 API마다 성공/실패 케이스 최소 1개 이상, @Valid 검증 포함"이라는 기준을 명문화하여 누구나 동일한 기준으로 테스트를 작성할 수 있도록 했습니다. 그 결과, 테스트 코드의 작성 방식이 표준화되었고, 실제 운영 환경과의 괴리를 줄여 테스트 신뢰도를 높일 수 있었습니다.
